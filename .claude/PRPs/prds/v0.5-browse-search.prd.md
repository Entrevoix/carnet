# PRD: carnet v0.5 — vault browse + search

**Status:** design, not scheduled · **Date:** 2026-07-04 · **Axis:** the "vault browse / cross-note linking" non-goal explicitly deferred by `v0.4-ai-deepening.prd.md` · **Source:** AUDIT.md §2, fact-pass over the current vault IO layer.

## Theme

Capture is solved (five modes, share target, widget, notification). Retrieval is not: the app can show the last 20 captures (`carnet:history:v1`, `HISTORY_LIMIT=20`) and browse by tag (`TagBrowserScreen`) — and that's all. "What have I been thinking about regarding X?" (`TODO.md` retrospective query) is blocked behind this. v0.5 makes the vault findable from the phone.

## Current state (verified)

- **Recents:** AsyncStorage list, 20 entries, insertion order, populated by `recordCapture` (`storage.ts:4,16,31`). Not a vault scan — a note older than 20 captures is unreachable in-app except through TagBrowser.
- **TagBrowser:** the only vault-wide surface. Backed by `buildTagIndex()` (`vault.ts:65`): `listNoteFiles()` then `readNote` on *every* note at concurrency 8 (`SCAN_CONCURRENCY`, `vault.ts:24`), cached in AsyncStorage `carnet:tagindex:v1`, invalidated fire-and-forget after every capture/drain/tag-edit.
- **Search code: none.** Greenfield. Only ranking precedent is prefix-then-substring tag suggestion (`vault.ts:159`, `tags.ts:34-39`).
- **Navigation seam already exists:** `synthesizeEntry(uri, md)` (`vault.ts:216`) adapts any vault note into a `CaptureEntry` for `RecentDetail` (history mutations no-op on `vault:` ids). Search results navigate exactly the way TagBrowser results already do. Deep-linking into RecentDetail is deliberately unsupported (`App.tsx:56`) — not a constraint for in-app search.
- **Hard constraint: no SQLite.** expo-sqlite@55 throws a SharedRef ABI error on-device against SDK 54 (`queue.ts:16-18`); the queue already retreated to AsyncStorage. Until the SDK upgrade lands, the index lives in AsyncStorage (like the tag index) — which also means no FTS5; ranking is done in JS.
- **SAF cost model:** one IPC read per note body, no batch-read API. Full-vault body scans are O(N) IPC round-trips; the tag index already pays this on rebuild, bounded at concurrency 8.

## Design

### Phase 1 — note-metadata index + Search screen

Generalize the proven tag-index pattern into a **note index**: one AsyncStorage blob (`carnet:noteindex:v1`), one entry per note:

```
{ uri, subdir, title, createdOrDate, tags[], mode, excerpt }
```

built from `listNoteFiles()` + the pure `frontmatter.ts` helpers (`parseFrontmatter`, `getFrontmatterTags`, `stripFrontmatter`) + `deriveTitle`. `excerpt` = first ~200 chars of the stripped body, captured during the same read that the tag scan already performs — **fold the tag index into this index** (tags are derivable from it) rather than maintaining two scans of the same files.

- Same lifecycle as today: cache-first read, pull-to-refresh rebuild, `invalidateNoteIndex()` at every write site that currently calls `invalidateTagIndex()`. Incremental upsert on capture (we know the URI and content we just wrote — no rescan needed for the common path).
- **Search screen** (`Stack.Screen "Search"` + `RootStackParamList` entry; entry point = a search icon in the HomeScreen header next to the existing TagBrowser/Settings icons, `HomeScreen.tsx:40-52`): single text field, results ranked prefix-then-substring over `title`, then `tags`, then `excerpt`, filter chips for mode/subdir, newest-first tiebreak. Result tap → `synthesizeEntry` → `RecentDetail`.
- Size check: 1,000 notes × ~300 bytes ≈ 300 KB JSON — fine for AsyncStorage; parse cost is one-time per app session (hold in memory after first read).

### Phase 2 — on-demand full-text search

For queries that miss the index, an explicit "Search note contents" action (not automatic) streams matches: `listNoteFiles()` → `readNote` at concurrency 8, case-insensitive substring over the stripped body, results appended as they arrive with a progress indicator, cancellable. No body cache in v1 of this phase — SAF read cost is paid per explicit request, which matches how rarely body search should be needed once titles/tags/excerpts are indexed.

### Phase 3 — retrospective query (bridges to v0.4 S4)

"What have I been thinking about X?" = Phase 1/2 retrieval feeding an existing enrichment-style LLM call that synthesizes an answer with `[[wikilinks]]` to sources. If v0.4 S4 (embeddings) ships, its vectors upgrade retrieval from substring to semantic; Phase 3 is designed against the *retrieval interface* (query → ranked note URIs), not against either implementation. Ship Phase 3 after S4's fate is known; do not build embeddings inside this axis.

## Non-goals

- Editing anything about search from the workstation side — Obsidian already owns that.
- Semantic/embedding search as a Phase 1/2 requirement (S4's axis; see Phase 3).
- Deep-link `carnet://search` — add only if a launcher-shortcut use case shows up; params are trivially url-encodable if so.
- Replacing TagBrowser — it becomes a filtered view of the note index (tag chip pre-applied), not a separate scan.

## Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Index rebuild slow on 1000+ note SAF vaults | Medium | Same exposure the tag index already has; incremental upsert makes rebuilds rare; concurrency 8 bound proven |
| AsyncStorage blob growth (multi-MB vaults) | Low-Medium | Metadata-only (~300 B/note). **Caveat:** Android AsyncStorage has a default **~6 MB total** limit *shared* across the queue + history + tag index + this note index — not per-key. At 5k+ notes the note index alone (~1.5 MB) plus siblings can approach that ceiling sooner than the per-index figure suggests. Mitigate: fold the tag index in (one blob, not two), cap the excerpt, and raise `AsyncStorage_db_size_in_MB` (gradle) if headroom is needed. In-memory parsed size (300 KB–1.5 MB) is fine on-device. |
| Index drift vs workstation edits arriving via Syncthing | Medium | Already true for the tag index; pull-to-refresh is the documented recovery; a mobile file-watcher is the TODO'd v0.3+ "bidirectional sync awareness" item — out of scope here |
| SDK upgrade un-breaks sqlite mid-build | Low | Index module isolated behind `noteIndex.ts` API; swapping the store later is internal |

## Open decisions

- **Fold-in vs parallel index:** recommendation above folds the tag index into the note index (one scan, one cache). The alternative — leave `carnet:tagindex:v1` untouched and add a second index — doubles scan cost but is zero-risk to TagBrowser. Decide at plan time after reading TagBrowser's coupling to the index shape.
- **Excerpt in index vs title/tags only:** excerpt adds ~60% blob size for substantially better hit rates on half-remembered notes. Recommended in; drop if the 5k-note math changes.
- **Journal granularity:** index journal files whole (one entry per day-file) or per `## HH:MM` block? Whole-file is Phase 1; per-block only if day-file hits prove too coarse in practice.
