# PRD: carnet v0.4 — AI deepening

## Theme
v0.3 made carnet a fully bidirectional capture surface: text, photo, audio, transcription, edit, persistent shortcuts. v0.4 doubles down on the LLM/OmniRoute infrastructure that makes captures *more useful*. Less new surface, more intelligence on existing surfaces.

## Goals
1. Reduce the per-capture user effort needed to get a *polished, searchable, well-tagged* note.
2. Keep the offline-first, local-first thesis. No cloud sync, no embeddings shipped to a third party beyond OmniRoute.
3. Stay Android-first. iOS is deferred to v0.5+.

## Non-goals
- Vault browse / cross-note linking — separate axis, deferred to v0.5
- iOS parity — deferred
- Continuous always-listening capture — too ambitious for v0.4; revisit later
- New file types beyond what v0.3 supports (text / image / audio / file shares)

## Slates

### Slate 1 — Auto-transcribe at save (S1)
**Effort:** ~1 PR, ~2 hours of work
**Confidence:** 9/10

Remove the "tap Transcribe" step for users who always want transcripts. AudioCaptureScreen and the shared-audio receive path get a post-save hook that fires transcribeAudio + upsertSection automatically.

**Scope:**
- New Settings toggle: `autoTranscribeOnSave: boolean`, default `false`
- In AudioCaptureScreen.stopAndSave: after `writeIdea` success, if the toggle is on, fire `transcribeAudio` + `upsertSection` + `updateNote`. Non-blocking — the saved screen still appears immediately, with an inline "Transcribing…" indicator that resolves into the transcript.
- Same hook in ShareReceiveScreen's audio branch.

**Risk:** doubles the OmniRoute spend per audio capture for users who flip it on. Documented + opt-in.

**Why first:** smallest scope, biggest UX win per LOC. Sets up the pattern S2 builds on.

---

### Slate 2 — Transcript polish pass (S2)
**Effort:** ~1-2 PRs, ~4-6 hours
**Confidence:** 7/10 (depends on prompt iteration)

Raw Whisper output is verbatim and unpunctuated. A second LLM call cleans it up — adds paragraph breaks, fixes obvious mishearings, generates a 2-3 sentence summary at the top.

**Scope:**
- New `polishTranscript(rawText)` in `omniroute.ts` — chat completion with a polish-specific prompt.
- 6th prompt-override mode in Settings: `transcriptPolish`.
- Polish button in RecentDetail (next to Transcribe), gated on the note having a `## Transcript` section already.
- Optional: auto-polish setting that chains S1 → polish (single tap → raw → polished, in two passes).
- Markdown shape: replace the `## Transcript` section's body with `## Summary\n...\n\n## Transcript\n...` (or add a `## Summary` section above and keep `## Transcript` as-is — TBD in plan).

**Risk:** polish prompts are tricky — too aggressive and you lose details, too light and nothing changes. Will need a few iterations on the default prompt before it ships well.

**Why second:** natural extension of S1, reuses the upsertSection helper.

---

### Slate 3 — Vault tag awareness (S3)
**Effort:** ~2 PRs, ~6-8 hours
**Confidence:** 6/10 (SAF tag-scan is the unknown)

Today carnet's auto-tagging creates tags freely. A user with a curated vault tag vocabulary gets sprawl: `dev` vs `development` vs `engineering`. Fix: at capture time, read existing tags from the vault and thread them into the LLM prompt as a hint ("prefer these tags if relevant").

**Scope:**
- New `scanVaultTags(): Promise<string[]>` in writer.ts — walks the vault via SAF, reads frontmatter `tags:` arrays from every `.md`, dedupes, returns sorted.
- Tag list is cached (5 min TTL or invalidate on writeIdea/appendJournal).
- All five enrich entry points (`enrichIdea`, `enrichJournal`, `enrichPerson`, `enrichSharedImage`, `enrichSharedLink`) thread the tag list into a new `availableTags` prompt variable.
- Prompts updated: instructions become "use these tags if relevant; else create new ones".
- Settings: `useExistingTagsForAutoTag: boolean`, default `true`.

**Risk:** SAF directory walks are slow on large vaults. 1000-note vault could take 5-10 seconds to scan. Mitigation: aggressive caching + background pre-warm on app launch.

**Why third:** independent of S1/S2, valuable on its own, but the SAF-scan complexity is real.

---

### Slate 4 — Find related notes (S4)
**Effort:** ~3 PRs, ~10-12 hours
**Confidence:** 5/10

Show "Related notes" on RecentDetail. Embed each capture, store the vector in local SQLite, compute cosine similarity against other recent captures, surface the top-3.

**Scope:**
- New `embedText(content): Promise<number[]>` in `omniroute.ts` — POSTs to `/v1/embeddings`.
- New `apps/mobile/src/lib/embeddings.ts` — SQLite-backed vector store (expo-sqlite is in deps).
- New `findRelated(filepath): Promise<RelatedNote[]>` — cosine-sim across stored vectors.
- New "Related notes" card on RecentDetail — tap to open (only works for entries in the 20-recents window; full vault navigation deferred to v0.5).
- Background job at save time embeds the new note + stores vector.
- Migration: backfill embeddings for existing recents on first launch with this feature.

**Risk:** embeddings work is novel ground for carnet. Cosine sim with 20 vectors is trivial. The novel parts: SQLite schema, background embed-on-save, the "related" UX surface. Real exploratory work.

**Why last:** highest scope, highest risk, depends on the others less than they depend on it. Could slip to v0.5 if the v0.4 budget runs short.

## Sequencing

```
S1 (auto-transcribe)
  └→ S2 (polish pass)
       └→ ship v0.4-rc1

S3 (vault tag awareness)
  └→ ships independently, any time

S4 (find related notes)
  └→ ships last, can be deferred to v0.5
```

**Recommended order:** S1 → S2 → S3 → S4

S1+S2 are the highest UX-per-LOC pair. S3 is a separable win whose value scales with vault size (rewards heavy users). S4 is the ambitious one — ship if budget allows, defer otherwise.

## Open decisions
- **S1 default:** off (opt-in) vs on (default toggle = true)? Suggest off — doubling API spend per audio capture is a real cost users should consent to.
- **S2 markdown shape:** replace `## Transcript` body with `## Summary + ## Transcript` (one section becomes two), or add `## Summary` above + leave `## Transcript` untouched? Suggest the latter — preserves the raw transcript as-is, summary lives above.
- **S3 scan trigger:** on every capture (with cache) vs only on app launch? Suggest both — launch warms the cache, subsequent captures reuse.
- **S4 embedding model:** `text-embedding-3-small` (cheap, 1536-dim) vs `text-embedding-3-large` (better, 3072-dim, ~6x cost)? Suggest small — quality is fine for recents-only similarity.

## Risks
| Risk | Severity | Mitigation |
|---|---|---|
| OmniRoute proxy doesn't expose `/v1/embeddings` (S4) | High | Pre-flight check in Settings → "Embeddings unavailable on your proxy"; gracefully hide S4 features |
| Polish prompt produces worse output than raw on some accents/languages (S2) | Medium | Ship with conservative default prompt + Settings override; users can tune |
| SAF tag scan slow on 1000+ note vaults (S3) | Medium | Background warm + 5min cache + truncate to top-100 most-used tags |
| Auto-transcribe lengthens recording-mode UX (S1) | Low | Save screen returns immediately; transcription is post-save inline indicator |
| Embedding storage corrupts on app crash mid-write (S4) | Low | SQLite handles transactions; vectors are rebuildable from `.md` content |
